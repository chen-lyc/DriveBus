# 调试：数量关系是证据，不是背景噪声

> 契约：未来读者重新打开时，应能在 30 秒内把日志中的“恰好、相差、周期、倍数”映射为可验证的容量、代际或单位假设；这篇笔记不承担任一 bug 的完整代码复盘。

## 核心模型

日志里的单个数字通常只是现象；**数字之间稳定的关系**才可能是证据。

```text
观察到数量关系
    ↓
找到代码中的容量 / 单位 / 周期常量
    ↓
写出能产生该关系的状态转换
    ↓
用代码映射或局部状态证实或推翻
```

数字关系只能产生一个可证伪的假设，不能单独宣布根因。它变成证据，至少还需要回答：这个数字对应哪个对象？单位相同吗？代码怎样把这个对象映射到日志中的值？

## 优先留意的数量模式

| 日志模式 | 首先怀疑 | 还要证明什么 |
| --- | --- | --- |
| 恰好处理 `N` 次后崩溃或异常 | 容量边界、off-by-one、绕回遗漏 | `N` 是否等于数组/队列容量；第 `N+1` 次实际访问哪个下标 |
| `actual - expected = N` | 一整批、一次重试、一个环的**物理周期**或一代对象被复用 | `N` 是否等于批量/物理槽位数；两者是否映射到同一物理位置 |
| 值相差 `sizeof(T)` 或按该比例缩放 | 字节数与元素数混用 | 参与比较、取模、下标的每个值的单位 |

## 两个 desc_ring 例子

### 1. 恰好 16 条后崩溃：容量边界

旧事件中，读端每次都恰好处理 16 条消息后，在第 17 条崩溃。`16` 与 `desc_ring` 容量一致，因此首先检查索引的绕回和单位。最终确认 `sizeof(desc_ring)` 的字节数被误当成元素个数，导致写端越过第 16 个槽位。

完整事件见：[desc_ring 越界 — sizeof vs 元素个数](01-desc-ring-oob-sizeof.md)。

### 2. `expected = 45`、`actual = 61`：一整圈 descriptor 复用

【本项目故障快照】下面描述的是产生该日志的当时构建；当前源码可能已经变化，不能据此断言当前版本仍含相同根因。

```text
read 11 byte, offset is 976, seq is 61
error seq: expected is 45, actual is 61
```

`kDescriptorSlotCount = 16` 是环的物理槽位数和取模周期；若采用留一格设计，可用容量是 15。这里的差值 `16` 指向“同一物理槽位的一整次复用周期”，不是“可用容量”。

故障构建初始化时 `seq = 0`、`wr = 0`，且每写一个 descriptor，`seq` 恰好递增一次。因此在该运行中：

```text
slot(seq) = seq % 16

45 % 16 = 13
61 % 16 = 13
61 - 45 = 16
```

`expected = 45, actual = 61` 的差值恰好是一整圈，立刻给出假设：seq61 复用了 seq45 的物理 descriptor 槽 13。

| 要证实的主张 | 本次证据 | 成立条件 |
| --- | --- | --- |
| `45` 和 `61` 对应同一槽 | 先有 `61 - 45 = 16`；每写一个 descriptor，`wr` 前进一格 | 相差一个物理槽位周期就必回到同一槽；精确槽号 13 还依赖本运行从 `seq=0, wr=0` 同相位开始 |
| `offset = 976` 属于 seq61 这一代 | 故障构建强制消息大小落在 8–16 字节，`find_size_class` 因而选 class0；class0 初始 head 从 0 开始，依次经过 64 个 16 字节 chunk，所以 seq45 为 720、seq61 为 976 | 每条消息都分配一个 class0 chunk；前 64 次仍沿初始空闲链分配，即使读端已归还 chunk，也尚未走到链尾 |
| 是 descriptor 而非 payload 被复用 | 读端先从**本地 descriptor**取 `offset`，再用该 offset 读取 payload；错误行已经显示新的 976 | 本地 descriptor 复制发生在这次读之前 |

因此，差值 16 先指出“一整圈复用”；`slot(seq)` 和 `offset` 再把它提升为“读端拿到 seq61 那一代的 descriptor”，而不是“seq45 的 descriptor 指向了后来被改写的 payload”。

数量关系没有直接告诉我们写端为什么越权；它把排查范围收窄为“写端为何允许同一 descriptor 槽跨一圈复用”。之后才检查环的最慢读端选择、满/空不变量和写入准入条件。

## 使用边界：不要把巧合当证明

- `N` 也可能是批大小、重试间隔、测试数据周期或日志采样周期；先列出所有同量级常量。
- 比较前先写单位：`16 个 descriptor`、`128 字节`、`16 条消息`不是同一个量。
- 只有“日志关系 → 代码映射 → 可导致该关系的状态转换”三段都成立，才把它写进根因链。

## 30 秒回顾

1. `actual - expected == 物理槽位数` 时，下一步应该验证什么？它为什么不是“可用容量”？
2. “恰好 N 次后失败”为什么不能直接等于“数组越界”？
3. 如何区分 descriptor 被复用与 payload 被复用？
