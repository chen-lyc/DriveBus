# 调试：并发读写 Bug 的证据链

> **简介**：未来的我再次遇到共享内存的 `magic`、`seq`、`no node free`、`kill -9` 后卡住，或“看起来卡在 `recv`”时，只用本页、当前日志、当前源码和调试器，不问 AI、不联网，应能先写出：精确事实、数量/单位关系、不变量、状态—写者表、两个可证伪假设、最短时序，以及一个能区分假设的验证动作。本页不替代某一次事故的完整根因记录，也不把跨进程日志的打印顺序当成真实全局时序。

**范围快照**：本页基于 DriveBus 的双读端、单写端改造过程整理；最后核对时的提交是 `36b7b0f`（2026-07-19）。历史现象对应的故障构建可能早于当前源码，不能把历史根因直接断言为当前仍存在的问题。

相关笔记：[数量关系是证据，不是背景噪声](02-debugging-numeric-patterns.md)讲”数字如何缩小范围”；[payload 头部魔数被 free list 覆盖](03-magic-overwritten-by-free-list.md)记录其中一次完整事故。本页只补足”如何把现象推到根因”的通用路线。

**三句话**：并发 Bug 不要从症状直接跳到修复——先把现象拆成 `[F]` 精确事实，写出不变量和状态—写者表，再做可证伪假设。数字关系和魔数分析是第一层筛选。`kill -9` 后的异常说明恢复路径参与了状态转换，不是独立 Bug。

## 先记住结论：高级调试不是更会猜

并发 Bug 常有多个表面现象：读端 `magic` 错、写端没有空闲节点、幸存读端超时、`kill -9` 后才出错。它们不必各自对应一个根因。

高级工程师的工作不是立刻报出答案，而是不断排除不能解释**全部关键事实**的路径，直到剩下一条可以复查的因果链：

```text
固定快照
→ 拆成精确事实
→ 找数字与单位关系
→ 写不变量
→ 列状态和写者
→ 展开最短时序
→ 提出可证伪假设
→ 用工具做区分性验证
→ 写清结论边界
→ 增加回归测试
```

`debugging` 是 `debug`（排除程序错误）加 `-ing`（进行中的过程），这里译为“调试”。`evidence chain` 由 `evidence`（证据）和 `chain`（链）组成，译为“证据链”。它要求相邻两步之间都有能解释的桥梁，不能从症状直接跳到修复。

| `timeline` | `time`（时间）+ `line`（线） | 把并发动作展开后的最短时序线。 |
| `[F]` / `[H]` / `[V]` / `[C]` / `[U]` | `Fact`（事实）/ `Hypothesis`（假设）/ `Verification`（验证）/ `Conclusion`（结论）/ `Unknown`（未知） | 强制区分已知、猜测、验证动作、已确认结论和未检查项。 |

### 本例必须认识的对象

| 名称 | 英文拆解 | 本例中负责什么 |
| --- | --- | --- |
| `magic` | `magic number`：魔数/固定校验值 | payload 开头的固定值；读端用它快速发现头部被意外改写。 |
| `descriptor` | descriptor 是“描述项” | 不保存消息内容本身，只保存 `offset` 和 `len`，告诉读端到共享 payload 的哪里、读多少字节。 |
| `payload` | payload：有效载荷 | 真正的消息字节；descriptor 即使复制到本地，它指向的 payload 仍在共享内存。 |
| `chunk` | chunk：一块/一段 | 分配器一次借出和归还的内存块；一个 payload 放在一个 chunk 内。 |
| `free list` | `free`（空闲）+ `list`（链表） | 把空闲 chunk 串起来的链；它会复用空闲 chunk 的开头保存 `next_offset`。 |
| `slot` | slot：槽位/固定位置 | 读端在共享数组中的稳定编号，不等于终端上“读端 1/2”的打印标签。 |
| reference bit / mask | `reference`（持有引用）+ `bit`（二进制位）/ `mask`（位掩码） | `1 << slot` 表示一个读端的持有位；mask 表示当前哪些读端仍持有 chunk。 |
| `size class` | `size`（大小）+ `class`（类别） | 按 chunk 大小划分的分配组；它决定相邻 chunk 的偏移间距。 |
| `SIGKILL` | `SIG` 是 `signal`（信号）的常用前缀，`KILL` 是强制终止 | `kill -9` 发送的不可捕获终止信号；读端没有机会自行完成清理。 |

## 这次最重要的工具：状态—所有权表

“读端已经读完”不是一个原子动作。它至少可能包含：复制 descriptor、读取 payload、清自己的引用位、把最后一个 chunk 接回 free list、发布 `read_index`。这些转换的**具体顺序必须从当前快照的源码读出**，不能凭名称假定；`kill -9` 可以发生在任意相邻两步之间。

先列出关键状态，而不是一开始读完整个程序：

| 状态 | 它表示什么 | 谁能改变它 | 改变前必须成立什么 | 改变后会影响什么 |
| --- | --- | --- | --- | --- |
| `conn_fd` | 一个具体读端的连接身份 | 连接管理和死亡检测路径 | 这确实是要检查的那条连接 | 哪个读端需要清理 |
| `subscriber_slot_index` | 读端在共享数组中的固定编号 | slot 分配者 | 映射唯一、稳定，且被双方采用 | `read_index[slot]` 与引用位的身份 |
| `descriptor_read_indices[slot]` | 该 slot 已处理到哪里的 descriptor 游标 | 对应读端或死亡清理路径 | 操作对象确实是同一个 slot | 写端是否能复用 descriptor 槽 |
| `chunk_reference_counts[chunk]` 的某个 bit | 哪些读端仍持有该 chunk | 发布者登记、对应读端清位、死亡清理 | 这个 bit 确实属于要操作的读端 | chunk 是否可以归还 |
| free-list 链接 | chunk 已空闲、可再次分配 | 最后持有者/回收路径 | 所有 holder 都已释放 | payload 头部可被 allocator 元数据覆盖 |

本例的跨字段不变量是：

```text
conn_fd
  → subscriber_slot_index
  → descriptor_read_indices[slot]
  → chunk_reference_counts 中的 (1 << slot) bit

必须始终描述同一个物理读端。
```

各字段单独看起来合理，并不代表组合正确。若连接身份、游标和引用位指向了不同读端，死亡清理会在“正确的数组下标”上清理“错误的人”。

## 固定调试路线：每一步必须产出什么

| 步骤 | 先问自己什么 | 要写下的产物 | 本次的例子 |
| --- | --- | --- | --- |
| 1. 固定快照 | 哪个 commit、哪组编译宏、哪个启动顺序产生此日志？ | 故障范围；正常与 `kill -9` 的唯一差异 | 旧代码、当前代码和未提交实验不能混着解释。 |
| 2. 拆事实 | 哪个字段错？哪个字段仍对？ | `[F]` 清单，保留原始数值和单位 | `magic` 错，但同一 payload 的 `seq` 对。 |
| 3. 数量关系 | `actual` 与 `offset`、chunk 大小、ring 槽位数相差多少？单位相同吗？ | 一个可证伪的物理机制候选 | `15360 = 14336 + 1024`，指向 1024-byte chunk 的 next offset。 |
| 4. 写不变量 | 什么状态绝不能在什么动作之前改变？ | 一句能被违反的规则 | 仍有任何 holder bit 时，chunk 不得进入 free list。 |
| 5. 列状态与写者 | 谁能写这个值？谁拥有这条身份？ | 上面的状态—所有权表 | 连接、slot、读下标、引用位必须统一。 |
| 6. 展开时序 | 哪些“一个动作”实际是多个共享状态转换？ | 3～6 步的最短 `timeline` | 读 payload、清 bit、归还、发布游标、死亡清理必须分开；顺序以当前源码为准。 |
| 7. 对比假设 | H1 和 H2 各解释什么？各缺什么？ | `[H1]`、`[H2]` 和区分它们的观察 | “free list 覆盖”与“descriptor 整体复用”需要不同证据。 |
| 8. 工具验证 | 这个工具结果究竟要推翻或支持哪一个假设？ | 一个具体断点、监视点或结构化日志 | GDB 的 `bt` 先判断写端卡在 descriptor 等待还是空闲节点等待。 |
| 9. 写结论边界 | 哪些事实已闭环？还有什么没检查？ | `[C]` 与 `[U]` | 一次测试通过不是所有纳秒级交错都安全。 |
| 10. 回归 | 修复后必须重现哪种触发条件？ | 有失败判据的脚本或测试 | 两读端都先推进，再分别 `SIGKILL` 一个，并检查幸存者和写端跨过一整圈。 |

跨进程 `cout` 输出不是全局有序快照：它只能提供线索。日志中“先打印”的一行，不足以证明另一个进程的共享状态一定在它之前或之后发生改变。

## Worked example：从魔数错误到“过早归还”

以下不是“看到 magic 错就猜 free list”的模板，而是本次三段证据怎样连起来。

```text
[F] read 557 byte, offset is 14336, seq is 3
[F] error magic: expected is 21354532, actual is 15360
[F] 557 落入 1024-byte size class，且 15360 = 14336 + 1024
```

本故障构建中，payload 头部的布局是：

```text
offset + 0 ... sizeof(int) - 1          = magic
offset + sizeof(int) ... 2*sizeof(int)-1 = seq
空闲后的同一开头                           = free-list 的 next_offset
```

本次目标平台中 `sizeof(int) == 4`，所以 `next_offset` 覆盖 magic 所在的前 4 字节，而紧随其后的 `seq` 可以仍保持原值。这是“`seq` 对、magic 错”为什么指向局部头部覆盖的必要桥梁；它只描述本故障构建的布局，不是所有协议都必须这样组织。

第一步只得到候选：`15360` 很像该 chunk 的后继偏移，而不是随机脏数据。

```text
[H] 该 chunk 在当前读端仍需读取时进入了 free list；free-list 链接覆盖了 offset 处的前 4 字节。
```

接着必须验证两件不同的事：

1. allocator（内存分配器）是否真的把 next offset 写进 chunk 头；
2. 为什么一个仍被持有的 chunk 的 holder（持有者）集合会错误地变成空。

这两项都成立后，才可写：

```text
[C] chunk 被提前归还；free list 正常写链接，但写入时机违反了引用生命周期。
```

`seq` 仍正确很重要：它表明这更像“头部被局部改写”，而不是整个 descriptor 或整个 payload 被另一条消息完全替换。它缩小范围，但单独不构成证明。

还要避免倒置因果：当前读端先校验 `magic` / `seq`，失败就直接退出；后面的清引用位和发布读下标不会执行。因此“随后写端卡住”可以是这次数据错误留下未清状态的**次生现象**，不能反过来证明“读端一开始没有往前读”。某次运行究竟是哪一种，仍要回到该次状态和时序验证。

完整的数值和位图因果链见：[payload 头部魔数被 free list 覆盖](03-magic-overwritten-by-free-list.md)。若现象是 `expected seq` 与 `actual seq` 相差环的物理槽位数，先查：[数量关系是证据，不是背景噪声](02-debugging-numeric-patterns.md)，不要混淆 descriptor 复用与 payload 生命周期结束。

## Worked example：为什么 `kill -9` 能出现“卡住”和“提前归还”两种现象

普通双读端运行正常、只在 `kill -9` 后异常，首先说明：异常恢复路径参与了状态转换；它**不**自动说明“死亡检测函数一定错了”。

历史版本中，读端曾自行用 CAS 认领 slot；而生产者后来需要用 `conn_fd → slot` 映射定位死亡读端。若二者不是同一个分配事实，就可能出现：

```text
生产者记录：连接 A → slot 0，连接 B → slot 1
读端实际 CAS：连接 A → slot 1，连接 B → slot 0
```

**若**实际发生了这类身份错位，死亡清理按“连接 A → slot 0”操作时会碰到连接 B 的游标和引用 bit。精确时序不同，可能有两个看似相反的结果：

| 清理效果 | 可见现象 | 前提 |
| --- | --- | --- |
| 真正死亡读端的 bit/游标没有被清掉 | chunk 永远不能完全归还，写端最后可能等到 `no node free` | 清理的 slot 不属于死亡连接。 |
| 幸存读端的 bit 被错清 | 引用掩码提前变为 0，chunk 被归还，幸存者看到被覆盖的 magic | 清理的 slot 不属于死亡连接。 |

这说明为什么不能把“写端卡住”和“读端读到已归还节点”立刻当成两个独立 bug。先检验它们能否共同违反同一个身份不变量；没有实际的 `conn_fd` / slot 追踪记录前，不能把某次运行的 A/B 反转写成 `[C]`。

```text
[U] 本次未保存某次故障运行中 conn_fd → 生产者 slot → 读端实际 slot 的关联追踪；
    因此 A/B 是否真的反转，仍不能由旧日志单独证实。
```

当前读端和死亡清理都以 `fetch_and(~subscriber_reference_bit)` 清位。`fetch_and` 由 `fetch`（取回旧值）和 `and`（按位与）组成：它原子地返回清位前的 mask，并应用按位与。对**同一个 slot 的同一个 bit**重复清除是幂等的：第一次清掉它，之后再清不会误减其他 bit；但这不是身份错位的补救。若 slot 已经认错，幂等操作仍会稳定地清掉幸存读端的正确 bit。先证明“是谁的 bit”，再讨论“重复清是否安全”。

下面是一个**用于推理的假设时序**，不是对某次旧日志中 A/B 归属的倒推结论：

```text
[H] 真实归属为 A → slot 1、B → slot 0；某个 chunk 的 mask 初始为 0b11。

1. A 已清自己的 bit 1：0b11 → 0b01。
2. A 在发布其后续状态前被 SIGKILL。
3. 写端按错误映射把 A 当成 slot 0。
4. 死亡清理清 bit 0：0b01 → 0b00，并把 chunk 接回 free list。
5. B 仍按 descriptor 的 offset 读取 payload，看到 free-list 写入的头部。
```

另一条 `[H]` 是：A 死亡前尚未清自己的真实 bit 1；错误清理 bit 0 后，bit 1 残留，chunk 无法归还。它解释“没有空闲节点”的方向。两条 `[H]` 的区别不是凭感觉决定，而是要用 `[V]` 验证：在同一结构化事件中记录 `conn_fd`、生产者分配的 slot、读端实际采用的 slot、清除的 bit、清理前后的 mask 和 `read_index`。必要时对 payload 头设置 `watchpoint`，确认究竟是谁把 next offset 写入。

历史事实与当前映射要分开：`a00f1b1` 的读端确实自行 CAS 认领 slot；`274a2a6` 后，生产者在传递 fd 时一并传递 slot，读端接收该 slot。可用下面命令复查历史，而不要把当前源码倒灌成旧事实：

```bash
git show a00f1b1:a.cpp
git show 5f7f9c5:a.cpp
git show 274a2a6 -- a.cpp b.cpp
```

当前实现的静态锚点：生产者传递 slot 的 [b.cpp：`send_fd`](../../b.cpp#L40-L62)，读端接收并保存 slot 的 [a.cpp：`recv_fd`](../../a.cpp#L24-L50)，死亡清理的 [b.cpp：`reap_dead_subscribers`](../../b.cpp#L129-L153)。这些锚点证明当前代码的路线；它们不单独证明所有并发时序安全。

## 工具不是答案：每个工具只回答一个问题

| 工具 | 在本问题中应该问什么 | 它不能单独证明什么 |
| --- | --- | --- |
| GDB 的 `bt` | 进程现在实际停在什么循环、从哪里调用到这里？ | 另一个进程的完整历史时序，或根因。 |
| `watchpoint` | 谁在何处写了某个 `offset` 的前 4 字节？ | 那次写为什么违反生命周期。 |
| 结构化日志 | 同一次状态变化中的 `conn_fd`、slot、`read_index`、bit、offset 是否一致？ | 不同进程打印行的全局先后。 |
| `MSG_PEEK | MSG_DONTWAIT` 的 flags | `recv` 是否会为“没有数据”而阻塞？ | 对端为何被判死、每个错误码是否已处理。 |
| 回归脚本 | 已覆盖的 `SIGKILL` 触发下，修复后是否继续推进且没有明确错误？ | 所有纳秒级并发交错都正确。 |

`[F]` 当次调试记录中的 GDB 栈显示 `recv(..., flags=66)`；先算：`66 = MSG_PEEK(2) + MSG_DONTWAIT(64)`。

- `MSG` 是 `message`（消息）；`PEEK` 是“窥视”，查看但不消费 socket 中的数据；
- `DONTWAIT` 是 `don't wait`（不等待），使这次尝试非阻塞；
- 因此“程序卡在 `recv`”这个直觉被推翻：下一步应检查非阻塞错误分支和 fd 生命周期，而不是先讨论 socket 阻塞。

这是一条通用纪律：**先让工具反驳一个明确的判断，再追问下一个问题。**

## 学习反思：不足是过程没有外化，不是“不会调试”

本次已经表现出的能力不能丢掉：注意到 magic 像空闲链表偏移、质疑 `min_index` 与 `wr` 的关系、主动用 GDB 查看实际卡点、并纠正“读端没有继续读”的过度概括。这些都是正确的调试动作。

需要补的是把这些动作固定下来：

| 已有能力 | 尚未固化的动作 | 下次立即执行的替代 |
| --- | --- | --- |
| 能发现异常数字 | 未必立刻写出等式和单位 | 先写 `actual = ?`、`offset = ?`、`chunk_size = ?`，再算关系。 |
| 能怀疑生命周期 | 未必列出所有能写该内存位置的路径 | 写“状态—写者表”，至少列 payload 写入、归还、free-list 链接。 |
| 能意识到 `kill` 会引入时序 | 容易把“读端死亡”压缩成一个动作 | 分成死亡前已完成、死亡中断、死亡后回收三段。 |
| 会使用 GDB | 可能先开工具、后定义问题 | 调用前写一句：它要区分 H1 还是 H2。 |
| 能察觉一个解释不完整 | 容易把合理猜测当成已确认根因 | 强制留下 `[U]`；缺证据就不写 `[C]`。 |

这不是人格或能力判断。它是“直觉尚未变成可复用外部流程”的问题；填完下一节的定位卡，直觉就能被检查、复盘和迁移。

## 学习层：闭卷练习（尚未证明掌握）

先折叠本页、独立作答，再打开答案核对。答案出现于同一页时，只能算 `guided practice`（有提示练习），不算真正的闭卷掌握。

### 练习 1：事实、假设和验证

给定：`magic` 错、`seq` 对、`actual = offset + chunk_size`。

1. 分别写出两条 `[F]`、一条 `[H]`、一条 `[V]`；
2. 此时绝不能直接宣布什么结论？

<details>
<summary>核对要点</summary>

`[F]` 可以是“magic 实际值等于 offset 加当前 class 的 chunk 大小”和“seq 仍正确”；`[H]` 可以是“free list 覆盖了 chunk 头”；`[V]` 必须检查 free list 是否真的写头部、以及 holder bit 是否可能被提前清空。此时不能宣布“free list 一定写错了”或“根因已确定”。
</details>

### 练习 2：身份错位的迁移

把本例改名为：`worker_fd → worker_slot → worker_progress[slot] → lease_mask`。写出一条身份不变量，并说明一个 slot 错位为什么既可能造成资源不归还，也可能造成仍在使用的资源被过早归还。

<details>
<summary>核对要点</summary>

四个状态必须始终描述同一个 worker。错位时，死 worker 的 lease bit 可能残留，资源不能回收；也可能误清活 worker 的 bit，使 lease mask 提前为零而过早回收。
</details>

### 练习 3：GDB 结果怎样改变问题

GDB 显示 `recv(..., MSG_PEEK | MSG_DONTWAIT)`。这能否证明 `recv` 阻塞？下一条应检查什么？

<details>
<summary>核对要点</summary>

不能。`MSG_DONTWAIT` 表示本次 recv 不应因“暂时没有数据”阻塞；它只把问题收窄到非阻塞返回后的分支，例如错误码处理与 fd 生命周期。还不能据此宣布具体根因。
</details>

### 练习 4：把“死亡”展开成最短时序

假定生产者记录 `A → slot 0`、`B → slot 1`，但读端实际 CAS 结果反过来；某 chunk 初始 `mask = 0b11`。

1. 分别为“ A 已清真实 bit 后死亡”和“A 未清真实 bit 就死亡”写出不超过 6 步的时序；
2. 每一步标出 mask 的变化，并说明哪个结果会导致“过早归还”、哪个结果会导致“资源不归还”；
3. 写一条 `[V]`：怎样记录或断住 `conn_fd → slot → bit`，使两个 `[H]` 可以区分？

<details>
<summary>核对要点</summary>

若 A 真实是 slot 1，且已经清 bit 1：`0b11 → 0b01`；错误清理 slot 0 后：`0b01 → 0b00`，可能过早归还。若 A 尚未清 bit 1：错误清理 slot 0 后是 `0b11 → 0b10`，真实死亡 bit 1 残留，资源可能不能归还。评分重点不是背出这两个 mask，而是：时序是否拆开了读 payload、清 bit、归还、发布游标和死亡清理；`[V]` 是否在同一次事件关联记录连接身份、两个 slot 来源、bit 和游标，而不是只打印一句“reader dead”。
</details>

## 回归证据与边界

[`tests/test_two_subscribers_kill9.sh`](../../tests/test_two_subscribers_kill9.sh) 定义的合同是：两个读端都已消费消息后，向其中一个发送 `SIGKILL`（`Signal Kill`：强制终止信号），再要求写端和幸存读端各自跨过超过 16 个 descriptor 槽位的进度，并检查 `error magic`、`error seq`、`time out`、`chunk free twice`、`chunk error use again` 和越界诊断。默认 `CASE_COUNT=2` 时，脚本在两个独立 case 中各杀一次两个读端；其他 `CASE_COUNT` 值不一定覆盖两者。参数、调用方式和失败证据见：[双读端 `kill -9` 回归测试说明](../../tests/README.md)。

脚本**成功执行且保留了 commit、命令、退出状态和输出**时，才构成已覆盖交错的行为证据。本页当前只静态核对了脚本合同；它既不把脚本存在当成通过记录，也不把一次通过写成“所有并发交错均安全”的数学证明。若要稳定命中某两个原子操作之间的窗口，仍需要专门的测试钩子或更强的可控调度。
