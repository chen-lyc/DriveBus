# 调试：沿状态链倒查，不按模块猜

> 范围快照：最后核对提交 `36b7b0f`（2026-07-19）。本页只记录方法——如何从症状推到根因，不替代单次事故的完整复盘。
>
> 相关：[数量关系是证据](02-debugging-numeric-patterns.md) · [magic 被 free list 覆盖](03-magic-overwritten-by-free-list.md)

## 核心模式

**不从"哪个模块看起来最可疑"开始。从不变量开始。**

并发 Bug 常有一长串状态依赖。"观测到 X"不等于"执行 X 的那段代码错了"——X 发生的前提可能在前面的环节就断了。正确的顺序是：

```
① 写出不变量链：这个结果要走完哪几步才能达成？
② 从症状端逐环倒查：最后一步真的执行了吗？→ 上一步的输入对了吗？→ 再上一步？
③ 最小证据区分假设：看哪个值能排除最多错误路径
④ 验证
```

## 错误案例：从症状直接跳到模块

### 现象

写端卡在 `while (head_off == tail_off)`，日志连续输出 `no node free`。没有 `kill -9`。

```
idx 2 read_index is 4
candidate is 14
write 14 byte n, seq is 13
min_descriptor_read_index is 0
idx 1 read_index is 5
idx 2 read_index is 4
no node free
no node free
...
```

### 用户的推理（踩坑版本）

```
[F] head == tail → 没有空闲节点
[F] 节点由读端归还 → 一定是读端归还代码有问题
[X] 但读端归还代码之前验证过没问题 → 逻辑不成立，卡住了
```

问题在哪？**第二步跳过了状态链。**

"节点归还"不是一个原子动作。它的完整链路是：

```
写端建立引用掩码 → 每个订阅者清一次位 → 最后一个清位者归还节点 → tail 推进
```

`head == tail` 只告诉你 tail 没推进——但 tail 没推进可能是**链上任意一环断了**，不只是最后一环"清位归还"。

### 真正的根因

写端发布消息时构造引用掩码遍历的是 `conn_fds`，但 `conn_fds` 只有定义、从未被填充——循环不执行，`chunk_reference_mask` 始终为 0。发布时等价于 `fetch_add(0)`，不给任何读端设引用位。

读端清位时看到旧值是 0，永远不满足"我是最后一个引用者"→ 永远不归还 → tail 不动 → head == tail。

**Bug 不在归还代码，在"建立引用"这一步。** 但用户的推理链从症状直接跳到了模块，中间两步完全没检查。

## 正确方法：从症状倒查，逐环验证

同一组日志，正确的倒查路线：

```
[F] head == tail → tail 没推进
                    ↓
[V] tail 由"最后一个清位者"在归还时推进
    归还条件: previous_reference_count == 自己的位
    检查: 读端清位时 previous_reference_count 是什么？
    → 发现一直是 0，从未进入归还分支
                    ↓
[V] 清位前引用掩码的值取决于发布时的 fetch_add
    检查: 发布时 fetch_add 的参数 chunk_reference_mask
    → 发现 mask 始终为 0
                    ↓
[V] chunk_reference_mask 由遍历订阅者集合构造
    检查: 遍历的集合 conn_fds 是否被正确填充？
    → 未填充 → 根因
```

**每一步只验证当前环节**，不跳、不猜。倒查到哪一环断了，哪一环就是根因。

## 倒查的四个环节（以本项目为例）

每次遇到"节点没有归还"时，按这个顺序查：

```
1. 最后清位归还 (tail 推进)
   → tail 的值有变化吗？读端进入过归还分支吗？
   → 查: previous_reference_count 的值

2. 每个订阅者清位
   → 每个读端的引用位被正确清除吗？
   → 查: fetch_and 前后的 mask 值、位图是否正确

3. 写端建立引用
   → 发布时设置的掩码包含所有订阅者吗？
   → 查: chunk_reference_mask 的值、遍历的集合是否正确

4. 身份映射
   → slot、conn_fd、引用位描述的是同一个读端吗？
   → 查: 分配 slot 的路径 vs 使用 slot 的路径是否一致
```

从 `1 → 4` 倒查。1 没断就查 2，2 没断就查 3。**不要跳。**

## 和"从模块猜"的对比

| | 按模块猜 | 沿状态链倒查 |
|---|---|---|
| 第一步 | "哪个模块最近改过？" | "tail 为什么没推进？" |
| 第二步 | "读端归还代码有 bug" | "进入归还分支的条件满足了吗？" |
| 第三步 | 对着归还代码找错 | "引用掩码设对了吗？" |
| 第四步 | 卡住，"可代码之前验证过" | 根因在建立引用环节 |

同一组日志，两种路线走向完全不同的结论。按模块猜会让 broker 的嫌疑被"没 kill 所以 broker 不该干预"排除——但 broker 改变了订阅者注册和 event_fd 的数据流，旧验证覆盖不了新路径。

## 为什么这个模式反复出现

之前几次 debug 也掉进了同一个坑：

- **desc_ring 越界**：`segfault` → "mmap 大小有问题？" → 实际是 `sizeof` vs 元素个数
- **magic 被覆盖**：`magic != expected` → "free list 写错了？" → 实际是位图身份错位导致过早归还
- **broker bug**：`no node free` → "读端归还代码有问题？" → 实际是引用掩码未建立

**每次都是同一个错误形状：从现象跳到"谁负责最终的可见结果"，跳过中间的状态链。** 不是因为推理能力不够——是因为没有先写出不变量链再动手。

## 动手前的检查

下次遇到并发 Bug，先写三样东西再碰代码：

```
1. 不变量链: 这个正常结果要走完哪几步？
2. 从症状端倒查第一问: 最后一步输入对了吗？
3. 最小区分证据: 看哪个值能告诉我"断了哪一环"？
```

标示：

| 标记 | 含义 |
|------|------|
| `[F]` | 事实——日志、GDB 输出中直接观察到的 |
| `[H]` | 假设——"如果 X 成立，则应该观察到 Y" |
| `[V]` | 验证——要查哪个变量/断点来区分假设 |
| `[C]` | 已确认——三段证据都成立后 |
| `[U]` | 未确认——知道缺什么，但不编造 |

**一条规则先行：不要从"哪个模块可疑"开始。先写状态链。**
